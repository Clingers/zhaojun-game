## 二、逐模块审查

### 2.1 core/GameEngine.ts ✅

**优点**：
- 职责清晰：初始化、场景切换、资源访问
- 暴露 AudioManager 和 SaveManager 的 getter，便于外部访问
- 有 destroy 方法，防止内存泄漏

**问题**：
1. **类型不完整**：`GameEngine` 类的属性没有显式类型声明
2. **sceneManager 未使用**：`this.sceneManager` 只在 `getScenes()` 时使用，其他地方未引用
3. **全局变量引用**：`SceneManager` 通过 `(window as any).gameEngine` 注入，耦合度高

**建议**：
- 将 `AudioManager` 通过 BaseScene 构造函数注入，而非全局查找
- `sceneManager` 可以删除，只在初始化时使用

---

### 2.2 core/SceneManager.ts ✅

**优点**：
- 集中注册所有场景
- `getScene(key)` 提供场景查找

**问题**：
1. **硬编码场景列表**：每增加一章都需要修改此文件
2. **无懒加载**：一次性实例化所有场景，内存占用高

**建议**：
- 改为动态导入：`import('./scenes/PrologueScene').then(m => new m.default())`
- 或使用工厂函数延迟创建

---

### 2.3 core/AudioManager.ts ✅

**优点**：
- 分层设计：masterGain → layerGain → source
- 支持淡入淡出
- 音频缓存避免重复加载

**问题**：
1. **内存泄漏风险**：`activeSources` 在 `stop` 后删除，但 `audioBuffers` 永不清理
2. **无错误处理**：`decodeAudioData` 失败时只 warn，未触发回调
3. **空间音效未实现**：`setSpatial` 方法只注释，无实现

**建议**：
- 添加 `dispose()` 方法，用于清空缓存
- `loadAudio` 返回 `Promise<AudioBuffer | null>`，失败时返回 null
- 实现 `setSpatial` 或删除该方法

---

### 2.4 core/SaveManager.ts ✅

**优点**：
- 简单直接，localStorage 封装到位
- 支持导入导出

**问题**：
1. **无版本管理**：数据结构变更后无法兼容旧存档
2. **无校验**：`importSave` 只 try-catch，未验证字段完整性

**建议**：
- 添加 `saveVersion` 字段，load 时检查版本
- 使用 TypeScript 类型断言确保数据完整性：`data as SaveData`

---

### 2.5 store/gameStore.ts ✅

**优点**：
- Zustand + persist 配置正确
- 状态更新函数语义清晰

**问题**：
1. **未导出类型**：`GameSettings` 和 `GameState` 未导出，外部无法引用
2. **resetProgress 未持久化**：调用后不会更新 localStorage

**建议**：
- 导出类型：`export type { GameSettings, GameState }`
- `resetProgress` 调用 `persist` 的 `setItem` 或使用 `partialize`

---

### 2.6 store/collectiblesStore.ts ✅

**优点**：
- 收集品数据结构完整
- `unlock` 方法有去重逻辑
- `getProgress` 计算百分比

**问题**：
1. **硬编码收集品列表**：9个收集品硬编码在文件里，新增需要手动添加
2. **收集品路径硬编码**：`imageUrl` 写死，不符合数据驱动原则
3. **无类型导出**：`Collectible` 和 `CollectiblesState` 未导出

**建议**：
- 将收集品定义提取到 `src/data/collectibles.json`，运行时加载
- 导出类型供组件使用

---

### 2.7 scenes/BaseScene.ts ⚠️

**优点**：
- 提供通用功能：对话显示、转场触发、收集品解锁

**问题**：
1. **全局依赖**：`window.gameEngine` 查找 AudioManager，耦合严重
2. **`useCollectiblesStore` 在 Phaser 类中使用**：这是 React Hook，不能在 Phaser 类中调用！**这是严重错误** ⚠️
3. **`transitionTo` 触发 React 事件**：但 BaseScene 本身不决定转场目标，由子类决定，职责不清

**建议**：
- AudioManager 通过构造函数注入：`constructor(chapterId: string, audioManager: AudioManager)`
- `onCollectibleCollected` 改为回调参数传递，避免在 Phaser 类中调用 Zustand
- 子类调用 `this.transitionTo('nextChapter')` 时，由 BaseScene 发出事件，React 监听后执行实际切换

---

### 2.8 scenes/PrologueScene.ts ~ Chapter10Scene.ts ✅

**优点**：
- 结构一致，继承 BaseScene
- 资源加载路径清晰

**问题**：
1. **全部是空实现**：`setupInteractions`、`onInteraction`、`playAmbientAudio` 都是空
2. **重复代码多**：每个文件结构几乎一样

**建议**：
- 随着剧本开发逐步填充
- 考虑使用数据驱动：每个章节的 JSON 配置定义交互点，BaseScene 统一解析

## 七、代码审查结论

### 7.1 核心问题（必须修复）

1. **React Hook 不能在 Phaser 类中调用**：
   ```typescript
   // ❌ 错误：BaseScene.ts
   protected onCollectibleCollected(collectibleId: string) {
     const { unlock } = useCollectiblesStore.getState(); // 错误！
     unlock(collectibleId);
   }
   ```
   **修复方案**：改为事件触发，React 侧监听并更新 store。

2. **全局变量耦合**：
   ```typescript
   // ❌ 不推荐
   const gameEngine = (window as any).gameEngine;
   ```
   **修复方案**：通过构造函数注入依赖。

---

### 7.2 重要改进（建议修复）

1. **SceneManager 延迟加载**
2. **SaveManager 添加版本管理**

4. **添加 ErrorBoundary**
5. **导出所有公共类型**
